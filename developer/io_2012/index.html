Finding Accessibility Errors
  • My team works a lot with other web developers at Google, and we've noticed a couple of things.
  • First, it's that many -- and I'd even say most -- web apps *can* be made more accessible to users with disabilities. Assistive technologies are getting better and better at handling rich, interactive pages. New HTML5 and ARIA techniques can go a long way to bridge the gap.
  • But the sad thing is, it's the simplest errors that can cause the biggest problems for users. Something as simple as a text field that isn't labeled, or a button that isn't focusable, can make an application unusable.
  • So if simplest errors are causing problems, we should be able to test for them, right?

Two things we know to be true

Many web apps are not fundamentally inaccessible in design.


Simple errors can cause big problems for users!
  • Unfortunately, many of the available accessibility testing tools aren't great.
  • They might be years out of date, where they assume web pages are more like documents than interactive applications.
  • Some of them only work on specific platforms, or assume that the end user will be using a particular platform.
  • And many of them don't integrate with your existing workflow as a developer.
  • We've found that many developers already spend much of their time in Chrome's Developer Tools...so why not test accessibility there?
  • So we created the Accessibility Developer Tools.

Existing tools didn't meet our needs

  • Some recommend older techniques like accesskey, longdesc, noscript, over newer alternatives from ARIA or HTML5
  • Some only work well for static content, with limited support for web apps
  • Some don't integrate well with existing workflows
  • Some only work on one platform

  • ...That's why we created the Accessibility Developer Tools for Chrome!
  • Here's just some of the errors the Accessibility Developer Tools checks for.
  • This isn't a complete list, but it covers a lot of the basics.
  • It makes sure that images and form fields are labeled, it checks if UI controls are focusable, it checks whether ARIA roles are being used correctly. It calculates the contrast ratio for text to make sure it's contrasty enough. We also do some checks for video accessibility, including looking for captions and fallback content.
  • You'll notice that these are simple errors. We decided to focus on these kinds of errors because (as I said before) these are the kinds of things that can cause the biggest problems for users.
  • We also wanted to work on catching the kinds of errors that we felt we could detect consistently and correctly. These errors are also some of the easiest for developers to fix!
  • As work on this project continues we'll be expanding the kinds of errors we check for.

Advanced Accessibility Techniques

Custom user interfaces and ARIA

  • Native HTML5 links and controls are accessible by default, e.g.:
      <button>Click here</button>
      <input type=range></input>
      <textarea></textarea>
  • ARIA (Accessible Rich Internet Applications), now part of HTML5, allows you to build arbitrary custom controls and make them accessible, for example a ratings gadget from last year's talk:
      <div id="stars" class="star_rating" tabindex="0" role="slider" aria-valuemin="0" aria-valuemax="5" aria-valuenow="3">...</div>
  • You can find plenty of examples online for accessible buttons, dialogs, list boxes, combo boxes, and more. What's next on the horizon?

    We can use a simple rule to avoid inverting JPEG images:

    html {
      -webkit-filter: invert()
    }
    img[src*="jpg"] {
      -webkit-filter: invert()
    }

    Chrome Speech Recognition Extension API

    Any text box can be speech-enabled with a single attribute, like this:

    <input x-webkit-speech>
    • Chrome also provides a more advanced app/extension API for speech recognition
    • This API is still experimental - the details may change

    Speech Recognition API example

    function checkStart() {
  if (chrome.extension.lastError) {
    alert("Couldn't start speech input: " + chrome.extension.lastError.message);
  }
}
function recognitionFailed(error) {
  alert("Speech input failed: " + error.code);
}
function recognitionSucceeded(result) {
  alert("Recognized '" + result.hypotheses[0].utterance +
        "' with confidence " + result.hypotheses[0].confidence);
}
chrome.experimental.speechInput.onError.addListener(recognitionFailed);
chrome.experimental.speechInput.onResult.addListener(recognitionSucceeded);
chrome.experimental.speechInput.start({ "language": "en" }, checkStart);

    Chrome TTS Engine API

    Users can install new text-to-speech voices with just a click!

    var speakListener = function(utterance, options, sendTtsEvent) {
  sendTtsEvent({'event_type': 'start', 'charIndex': 0})
  // (start speaking)
  sendTtsEvent({'event_type': 'end', 'charIndex': utterance.length})
};
var stopListener = function() {
  // (stop all speech)
};
chrome.ttsEngine.onSpeak.addListener(speakListener);
chrome.ttsEngine.onStop.addListener(stopListener);

    TTS Engine implementation possibilities

    • The TTS Engine API just handles dispatching requests for speech to your extension - it's up to you how to actually generate the speech and output the audio.
    • What are some different ways to generate speech from a Chrome extension?
      • Send request to server, stream resulting audio
      • Pure-JavaScript using HTML5 Audio APIs
      • Our preferred solution: Native Client
